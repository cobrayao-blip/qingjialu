# 清嘉录 · 部署手册

本文档面向在 **Ubuntu Server 24.04 LTS** 上单机部署：域名 **`qjl.textengine.cn`**、公网 **`43.142.176.9`**、**PostgreSQL 数据目录位于独立 60GiB 数据盘**（与系统盘分离）。

**推荐方式**：使用仓库提供的 **Docker Compose 生产编排**（`docker-compose.prod.yml`），与本地 **`docker-compose.yml` 开发环境**同一套容器工具链，差异在于生产镜像内为 **构建后的静态资源 + `npm start` 跑 API**，无 Vite 热更新、无宿主机源码挂载。

**备选方式**：在宿主机直接安装 Node、PostgreSQL、Nginx（**裸机部署**），见本文后半部分。

> **约定**  
> - 数据盘挂载点：**`/data`**（请按云控制台实际设备名调整，文中以 `/dev/vdb` 为例）。  
> - Docker 方案：项目目录建议 **`/data/qjl/app`**；PostgreSQL 数据目录绑定 **`/data/qjl/pgdata`**（见 `.env.docker.prod.example`）。  
> - 裸机方案：应用系统用户 **`qjl`**，代码目录 **`/data/qjl/app`**；数据库用户/库名 **`qingjialu`**（可改，需同步 `.env`）。

### 你需要事先准备什么

| 准备项 | 说明 |
|--------|------|
| **云服务器** | 例如已购腾讯云 CVM，系统 **Ubuntu 24.04**，知道 **公网 IP**。 |
| **会登录服务器** | 用 **SSH**：在自己电脑上打开终端（Windows 可用 **PowerShell**），执行 `ssh ubuntu@你的公网IP` 或厂商提供的用户名（有时是 `root`）。 |
| **安全组 / 防火墙** | 在云控制台给这台机器放行 **22**（远程登录）、**80**（网站 HTTP）、**443**（HTTPS，后面要用）。**22 建议只对你家/公司 IP 开放**，不要对全世界开放。 |
| **域名（可选但强烈建议）** | 若有域名（如 `qjl.textengine.cn`），在域名 DNS 里加 **A 记录** 指向服务器**公网 IP**。没有域名时，可先用 `http://公网IP` 访问（浏览器会提示「不安全」属正常，后续再上证书）。 |
| **项目代码** | 服务器上能 `git clone` 你的仓库；若不会 Git，也可本机打包源码上传到服务器（需保证目录里含 `docker-compose.prod.yml`、`Dockerfile.prod`）。 |
| **一点耐心** | 第一次构建镜像可能较慢；命令报错时，把**完整报错**复制下来对照本文 [常见问题](#10-常见问题)。 |

### 第一次上线：建议操作顺序（Docker）

1. **SSH 登录服务器**。  
2. **安装 Docker**（若 `docker compose version` 提示找不到命令，见下文 [在 Ubuntu 上安装 Docker](#在-ubuntu-上安装-docker)）。  
3. **（可选）** 挂载数据盘到 `/data`：见 [数据盘分区与挂载](#2-数据盘分区与挂载)。没有单独数据盘时，也可先把 `PGDATA_HOST_PATH` 指到系统盘目录（如 `/var/lib/qjl-pgdata`），但生产更推荐独立云硬盘。  
4. **创建数据库目录并授权**，编辑 **`.env.docker.prod`**（见 [环境文件逐项检查](#环境文件逐项检查)）。  
5. **拉代码 → `docker compose ... up -d --build`**。  
6. **自检**：见 [部署完成后自检](#部署完成后自检)。  
7. **浏览器访问** `http://公网IP` 或域名，用 **`.env.docker.prod` 里的管理员账号** 登录。  
8. **HTTPS**：DNS 生效后再用 Certbot 或负载均衡证书（见 [HTTPS 与对外端口](#https-与对外端口)；纯新手可第二天再做）。

| 词 | 通俗含义 |
|----|----------|
| **SSH** | 远程登录到服务器里执行命令。 |
| **安全组** | 云厂商的「防火墙规则」，决定哪些网络端口能从公网访问你的机器。 |

| **环境变量 / `.env` 文件** | 存密码、密钥的配置，不给写进代码里；**不要**把 `.env.docker.prod` 发到公开地方。 |
| **80 / 443 端口** | 浏览器访问网站默认走 80（HTTP）和 443（HTTPS）。 |

## Docker 生产部署（推荐）

与仓库内 **`docker-compose.yml`** 的**开发模式**（Vite 热更新、源码挂载、`dev:server:watch`）不同，生产请使用 **`docker-compose.prod.yml`** + **`Dockerfile.prod`**：三服务 **db（PostgreSQL 16）**、**api（Express + tsx）**、**web（Nginx：内置 `dist` + 反代 `/api`）**。

### 架构与端口（Docker）

| 组件 | 说明 |
|------|------|
| **web** | **Nginx**，容器内 `80`；默认映射宿主机 **`${WEB_PORT:-80}:80`**（可在 `.env.docker.prod` 中设置 `WEB_PORT`，例如 `8080` 或 `127.0.0.1:8080`） |
| **api** | **Node**，仅容器网络内 **`3001`**，不对宿主机暴露（由 web 反代） |
| **db** | **PostgreSQL**，仅容器网络内 **`5432`** |

浏览器访问 **`http://服务器公网 IP`** 或域名；**不要**在前端构建时设置 `VITE_API_BASE`（同域 `/api`）。

**访问与账号**：站点为 **登录后使用**。打开首页会先进入用户登录；除 **`POST /api/auth/login`** 外，业务接口均需 **`Authorization: Bearer <JWT>`**（由前端登录后自动携带）。首次部署请在环境变量中配置 **`ADMIN_USERNAME` / `ADMIN_PASSWORD`**（见下文），用该账号登录后，可在「管理后台 → 用户管理」中创建 **viewer / editor** 账号。

**角色与界面（与当前前端一致）**：**viewer**（只读账号）登录后进入 **主站首页** 浏览内容；顶栏 **不显示「管理后台」**，若直接访问 **`/admin`** 会回到首页。**editor / admin** 可使用地理治理与校审工作台（`/admin`）；**大模型配置、用户管理** 仍仅 **admin**。

### 在 Ubuntu 上安装 Docker

在服务器上执行（**Ubuntu 24.04** 适用；整段可复制粘贴，来自 [Docker 官方文档](https://docs.docker.com/engine/install/ubuntu/) 的常用写法）：

```bash
sudo apt-get update
sudo apt-get install -y ca-certificates curl
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu $(. /etc/os-release && echo "${VERSION_CODENAME}") stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
sudo apt-get update
sudo apt-get install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
```

验证（应看到 **Compose** 版本号）：

```bash
sudo docker compose version
```

**（推荐）** 把当前用户加入 `docker` 组，之后不用每条命令都写 `sudo`（改完需**重新登录 SSH** 才生效）：

```bash
sudo usermod -aG docker "$USER"
```

若官方源较慢，可使用云厂商提供的 **Docker 安装镜像/文档**，或仅在「构建镜像」步骤使用手册中的 **`BASE_IMAGE` 国内镜像**（见 [镜像构建与国内源](#镜像构建与国内源)）。

### 前置条件

1. 已安装 **Docker Engine** 与 **Compose V2**（`docker compose version` 或 `sudo docker compose version`）。
2. 数据盘已挂载到 **`/data`**（见下文 [数据盘分区与挂载](#2-数据盘分区与挂载)）。
3. 在宿主机创建 PostgreSQL 数据目录。**`postgres:16-alpine`** 镜像内 **`postgres`** 用户 UID 一般为 **70**（若启动失败请用下方命令核对后再 `chown`）：

```bash
sudo mkdir -p /data/qjl/pgdata
sudo chown 70:70 /data/qjl/pgdata
docker run --rm postgres:16-alpine sh -c "id -u postgres"
```

4. 克隆项目至 **`/data/qjl/app`**（或任意目录），进入项目根：

```bash
sudo mkdir -p /data/qjl
sudo chown -R "$USER:$USER" /data/qjl
cd /data/qjl
git clone <你的仓库URL> app
cd app
cp .env.docker.prod.example .env.docker.prod
nano .env.docker.prod
```

### 环境文件逐项检查

编辑 **`.env.docker.prod`** 时，**至少**确认下表（缺一项都可能导致容器起不来或无法登录）：

| 变量 | 是否必填 | 说明 |
|------|----------|------|
| `PG_PASSWORD` | **必填** | 数据库 `qingjialu` 用户的密码，自己设强密码；与下面 `db` 容器使用同一值。 |
| `JWT_SECRET` | **必填** | 在服务器执行 `openssl rand -hex 32`，把输出**整行**贴进此变量（不要引号、不要空格）。 |
| `PGDATA_HOST_PATH` | 强烈建议 | 例如 `/data/qjl/pgdata`；目录需已 `mkdir` 且 **`chown` 为 postgres UID（多为 70）**（见前置条件）。 |
| `ADMIN_USERNAME` | **强烈建议** | 首次登录的管理员用户名，如 `admin`。 |
| `ADMIN_PASSWORD` | **强烈建议** | 管理员登录密码（强密码）。 |
| 密钥与密码 | 勿泄露 | `JWT_SECRET`、`PG_PASSWORD` 等不要截图发群、不要提交 git。 |

保存文件后可用 `cat .env.docker.prod` 快速检查是否还有中文占位符（如「改为强密码」未改掉会导致理解混乱，应换成真实密码）。

### 启动与更新

```bash
cd /data/qjl/app
docker compose -f docker-compose.prod.yml --env-file .env.docker.prod up -d --build
```

常用运维命令：

```bash
docker compose -f docker-compose.prod.yml --env-file .env.docker.prod ps
docker compose -f docker-compose.prod.yml --env-file .env.docker.prod logs -f --tail=200 api
```

发版拉代码后重新构建：

```bash
git pull
docker compose -f docker-compose.prod.yml --env-file .env.docker.prod up -d --build
```

### 部署完成后自检

在服务器上**按顺序**执行（假设已 `cd` 到项目根目录，且未把用户加入 `docker` 组时，在每条 `docker` 前加 `sudo`）：

```bash
# 1) 三个服务是否都是 Up / healthy
docker compose -f docker-compose.prod.yml --env-file .env.docker.prod ps

# 2) 本机能否访问 Nginx（web 映射了 80 时）
curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1/

# 3) 经 Nginx 反代的健康检查（应返回 JSON 含 ok）
curl -s http://127.0.0.1/health

# 4) 查看 API 最近日志（报错时先看这里）
docker compose -f docker-compose.prod.yml --env-file .env.docker.prod logs --tail=80 api
```

然后在**你自己电脑**的浏览器打开 **`http://服务器公网IP`**（或域名）。若打不开：先回云控制台确认 **安全组已放行 80**；再在服务器 `curl http://127.0.0.1/` 对比，判断是「服务器内部问题」还是「公网被挡」。

### 与开发用 Compose 的对应关系

| 文件 | 用途 |
|------|------|
| `docker-compose.yml` | **开发**：`web` 跑 Vite dev、`api` 跑 `tsx watch`，可挂载源码 |
| `docker-compose.standalone.yml` | 可选：去掉部分挂载，仍非生产镜像 |
| **`docker-compose.prod.yml`** | **生产**：多阶段构建、`npm start`、Nginx 托管静态资源 |

### HTTPS 与对外端口

- 默认 **`WEB_PORT`** 未设置时，映射为 **`80:80`**。若宿主机还需自建 Nginx 占 80，可在 `.env.docker.prod` 中设置例如 **`WEB_PORT=127.0.0.1:8080`**，由宿主 Nginx/Caddy 监听 **443** 并反代到 **`http://127.0.0.1:8080`**；或使用云上 **CLB/HTTPS 证书** 指向该端口。
- 容器内 Nginx 配置见 **`deploy/nginx.docker.conf`**（已含 **`client_max_body_size 100m`**）。

### 镜像构建与国内源

构建时可指定基础镜像（与根目录 `Dockerfile` 相同参数名）：

```bash
docker compose -f docker-compose.prod.yml --env-file .env.docker.prod build \
  --build-arg BASE_IMAGE=docker.m.daocloud.io/library/node:22-bookworm-slim
```

### 数据卷说明

- 在 `.env.docker.prod` 中设置 **`PGDATA_HOST_PATH=/data/qjl/pgdata`** 时，数据库文件在**宿主机数据盘**，便于快照与迁移。
- 若未设置 `PGDATA_HOST_PATH`，Compose 会使用命名卷 **`qjlpg_prod`**（适合本机试跑，**云上生产仍建议绑定 `/data/...`**）。

## 开发与生产数据

### 开发数据在哪？

- **正常运行时**（本机 `npm run dev:all` 或 **`docker-compose.yml` 开发**）：业务数据**全部在 PostgreSQL** 里（`server/db.ts` 使用 `pg`）。
- 使用 **`docker-compose.yml`** 时，PostgreSQL 的数据文件在 Docker 声明的**命名卷 `pgdata`** 里（见下一小节）。

### `pgdata` 卷是什么？

`docker-compose.yml` 里 **`db`** 服务有如下挂载（含义：把名为 **`pgdata`** 的 **Docker 命名卷**挂到容器内 Postgres 的数据目录）：

```yaml
volumes:
  - pgdata:/var/lib/postgresql/data
```

文件末尾还有：

```yaml
volumes:
  pgdata:
```

这表示：Docker 在宿主机上分配一块由引擎管理的存储空间（卷名 **`pgdata`**），里面才是 PostgreSQL 的表空间、WAL 等——**你开发的绘本、翻译缓存、解析结果等都在这块卷里**。在 Linux 开发机上可用 `docker volume ls` 看到 `qingjialu_pgdata` 或带项目前缀的卷名，用 `docker volume inspect <卷名>` 查看实际路径（通常在 `/var/lib/docker/volumes/...`）。  
它与 compose 里 **`./data:/app/data`** 挂载是**两回事**：后者主要是容器内应用目录下的 `data` 共享，**业务库默认不依赖它来存 PostgreSQL 数据**。

### 会不会自动进生产？会不会被删？

- **绘本、翻译缓存、解析/时令/地理缓存、图谱相关数据、会话、以及保存在库内的大模型配置等**，全部在 **PostgreSQL** 里；**不会**打进 Docker 镜像，也**不会**随 `npm run build` 的 `dist` 一起部署。
- **上线到生产**时，生产连接的是**另一套** PostgreSQL（云上新库、新数据目录或新命名卷）。应用首次启动会 **initDb 建表**，此时业务表是**空的**；**不会自动**把你开发环境里的数据复制过去。
- **开发环境的数据不会因为「部署生产」而被删除**：只要不手动删库、不删本机 Docker 卷，开发库仍留在原处；开发与生产是**两套库**，互不影响。

### 如何让生产也有同一批数据？

需要你在上线后**主动做一次逻辑备份 → 恢复**（或定期同步策略）。下面以 **Docker 开发 Compose** 与 **Docker 生产 Compose** 为例；若开发是裸机 PostgreSQL，把 `pg_dump` 改成指向本机 `localhost` 即可。

**1）在开发机导出（开发 compose 里数据库服务名为 `db`）**

```bash
cd /你的项目路径/qingjialu
docker compose exec -T db pg_dump -U qingjialu -Fc qingjialu > qingjialu_dev.dump
```

**2）把 `qingjialu_dev.dump` 拷到生产服务器**（`scp`、网盘等）。

**3）在生产库导入**（导入前建议**先停 API**，避免连接占用；生产 compose 同样使用服务名 **`db`**）

```bash
cd /data/qjl/app
docker compose -f docker-compose.prod.yml --env-file .env.docker.prod stop api
docker compose -f docker-compose.prod.yml --env-file .env.docker.prod exec -T db \
  pg_restore -U qingjialu -d qingjialu --clean --if-exists < /path/to/qingjialu_dev.dump
docker compose -f docker-compose.prod.yml --env-file .env.docker.prod start api
```

> **`--clean --if-exists`**：会先删掉目标库里与备份中**同名**的对象再导入，适合**用开发整库覆盖一台全新的生产库**。若生产上已经产生**不能丢**的新数据，不要用整库覆盖，应改为只导部分表或找专业人员做合并方案。

## 1. 架构与端口

| 组件 | 说明 |

| 浏览器 | 访问 `https://qjl.textengine.cn` |
| Nginx | 监听 **80/443**；提供 `dist` 静态文件；将 **`/api`** 反代到本机 **127.0.0.1:3001** |
| Node（Express） | 监听 **3001**（仅本机，不对公网暴露） |
| PostgreSQL | 监听 **127.0.0.1:5432**（建议仅本机） |

**DNS**：为 `qjl.textengine.cn` 配置 **A 记录** → **`43.142.176.9`**。内网 IP（如 `172.17.0.7`）仅用于 VPC 内互通，不要写入公网 DNS。

**同域部署**：前端构建时**不要**设置 `VITE_API_BASE`，由浏览器请求相对路径 `/api/...`，经 Nginx 转发到后端。

## 2. 数据盘分区与挂载

在云控制台将 **60GiB 数据盘**挂载到该 CVM 后，SSH 登录，确认块设备名称（常见为 **`/dev/vdb`**，也可能是 `vdc` 等）：

```bash
lsblk
```

以下假设数据盘为 **未格式化的空盘** `/dev/vdb`。若盘上已有分区，请改用对应分区设备（如 `/dev/vdb1`）。

```bash
# 格式化为 ext4（会清空该盘全部数据）
sudo mkfs.ext4 -L qjldata /dev/vdb

# 挂载点
sudo mkdir -p /data
sudo mount /dev/vdb /data

# 开机自动挂载（nofail：盘未挂上时仍可启动系统，按需选用）
echo 'LABEL=qjldata /data ext4 defaults,nofail 0 2' | sudo tee -a /etc/fstab
# 若未打 LABEL，改用设备 UUID 更稳妥：
# sudo blkid /dev/vdb
# echo 'UUID=你的UUID /data ext4 defaults,nofail 0 2' | sudo tee -a /etc/fstab

sudo mkdir -p /data/qjl /data/postgresql
sudo chmod 755 /data /data/qjl /data/postgresql
```

后续：**应用代码**放在 `/data/qjl/`，**PostgreSQL 数据目录**放在 `/data/postgresql/...`（见下一节）。

## 3. PostgreSQL：数据目录放到数据盘

Ubuntu 24.04 仓库中的 PostgreSQL 主版本一般为 **16**。以 **`pg_lsclusters`** 输出为准。

### 3.1 安装 PostgreSQL

```bash
sudo apt update
sudo apt install -y postgresql postgresql-contrib
pg_lsclusters
```

记下 **Ver**（版本号）与 **Name**（一般为 `main`）。下文以 **`16` / `main`** 为例，若版本不同请替换路径中的 `16`。

### 3.2 停库、迁移数据目录

**原则**：把集群数据目录从默认的 `/var/lib/postgresql/16/main` 迁到 **`/data/postgresql/16/main`**，并修改配置中的 `data_directory`。

```bash
sudo systemctl stop postgresql

# 数据盘上的目标目录（属主必须为 postgres）
sudo mkdir -p /data/postgresql/16
sudo rsync -aP /var/lib/postgresql/16/main/ /data/postgresql/16/main/
sudo chown -R postgres:postgres /data/postgresql

# 保留一份备份，确认新目录可启动后再删
sudo mv /var/lib/postgresql/16/main /var/lib/postgresql/16/main.bak.$(date +%Y%m%d)
```

编辑集群配置中的 **`data_directory`**（路径以实际版本为准）：

```bash
sudo sed -n '1,80p' /etc/postgresql/16/main/postgresql.conf | grep data_directory
```

将其中一行改为（若没有则增加）：

```conf
data_directory = '/data/postgresql/16/main'
```

也可用编辑器打开 `/etc/postgresql/16/main/postgresql.conf` 全文搜索 `data_directory` 后修改。

**可选（推荐）**：若希望 PostgreSQL 只在本地回环监听，在同一文件中确认或设置：

```conf
listen_addresses = 'localhost'
```

启动并验证：

```bash
sudo systemctl start postgresql
sudo systemctl status postgresql
sudo -u postgres psql -c "SHOW data_directory;"
```

应显示 **`/data/postgresql/16/main`**。

确认无误后，可删除旧数据备份以释放系统盘空间（**谨慎**）：

```bash
sudo rm -rf /var/lib/postgresql/16/main.bak.*
```

### 3.3 新建业务用户与数据库

```bash
sudo -u postgres psql -c "CREATE USER qingjialu WITH PASSWORD '此处改为强密码';"
sudo -u postgres psql -c "CREATE DATABASE qingjialu OWNER qingjialu;"
sudo -u postgres psql -c "GRANT ALL PRIVILEGES ON DATABASE qingjialu TO qingjialu;"
```

应用 `.env` 中的 `PG_USER`、`PG_PASSWORD`、`PG_DATABASE` 需与这里一致。

### 3.4 （备选）用 bind mount 不改 `data_directory`

若不想改 `postgresql.conf`，可在迁移数据后，将原路径挂载为数据盘子目录的绑定挂载：

```bash
# 示例：数据在 /data/pgdata/main，仍使用默认配置路径
sudo mkdir -p /data/pgdata/main
sudo rsync -aP /var/lib/postgresql/16/main/ /data/pgdata/main/
sudo rm -rf /var/lib/postgresql/16/main
sudo mkdir /var/lib/postgresql/16/main
echo '/data/pgdata/main /var/lib/postgresql/16/main none bind 0 0' | sudo tee -a /etc/fstab
sudo mount -a
sudo chown -R postgres:postgres /data/pgdata/main /var/lib/postgresql/16/main
sudo systemctl start postgresql
```

两种方式二选一即可；**推荐 3.2**（`data_directory` 显式指向 `/data`），路径清晰、便于备份说明。

## 6. Nginx 与 HTTPS

### 6.1 站点配置

创建 **`/etc/nginx/sites-available/qjl.textengine.cn`**：

```nginx
server {
    listen 80;
    server_name qjl.textengine.cn;
    client_max_body_size 100m;

    root /data/qjl/app/dist;
    index index.html;

    location /api/ {
        proxy_pass http://127.0.0.1:3001;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 300s;
    }

    location /health {
        proxy_pass http://127.0.0.1:3001/health;
        proxy_set_header Host $host;
    }

    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

启用并重载：

```bash
sudo ln -sf /etc/nginx/sites-available/qjl.textengine.cn /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl reload nginx
```

### 6.2 Let’s Encrypt

**在 DNS 已生效指向本机公网 IP 后**执行：

```bash
sudo certbot --nginx -d qjl.textengine.cn
```

Certbot 会改写配置增加 **443**。请检查 **443** 的 `server { ... }` 块内仍包含 **`client_max_body_size 100m;`**（若缺失，手动补上，否则大请求可能被 Nginx 拒绝）。

## 10. 常见问题

### 10.1 请求 413 Payload Too Large

后端已放宽 JSON body；若仍 413，多为 **Nginx** 未设置或 **443** 站点未继承 **`client_max_body_size`**。参见仓库 **`docs/413-payload-too-large.md`**。

### 10.2 前端能打开但接口失败

- **裸机部署**：`systemctl status qingjialu-api` 与 `journalctl -u qingjialu-api -n 100 --no-pager`  
- **Docker 部署**：`docker compose -f docker-compose.prod.yml --env-file .env.docker.prod logs --tail=100 api`；在 **api** 容器内自测：`docker compose ... exec -T api curl -s http://127.0.0.1:3001/health`  
- 本机直连 API（裸机或已映射端口时）：`curl -s http://127.0.0.1:3001/health`  
- 裸机 Nginx：检查 `location /api/` 的 `proxy_pass` 是否指向 **3001**

### 10.2.1 接口返回 401「未登录或会话已失效」

- 属预期行为：**未登录或 JWT 过期**时，除登录接口外的 `/api/*` 会返回 401。请在网页上重新登录。  
- 若已登录仍大量 401：检查浏览器是否禁用 **localStorage**、是否清除了站点数据；核对 **`JWT_SECRET`** 是否曾被更换（更换后旧 token 全部失效，需重新登录）。  
- 直连调试 API 时须自行加请求头：`Authorization: Bearer <登录接口返回的 token>`。

### 10.2.2 无法登录 / 无可用账号

- 确认 `.env` 或 `.env.docker.prod` 中已设置 **`ADMIN_USERNAME`、`ADMIN_PASSWORD`**，且密码与预期一致；或向库中已有用户索取账号（角色为 **admin / editor / viewer** 均可登录主站）。  
- **大模型配置、用户管理**等仍仅 **admin** 可用；**viewer / editor** 由 admin 在后台创建。  
- **viewer** 登录后只在主站浏览；**无后台入口**、**不可停留在 `/admin`**（会自动回首页）。若期望某账号能进工作台，请使用 **editor** 而非 viewer。

### 10.3 PostgreSQL 启动失败

- `sudo journalctl -u postgresql -n 80 --no-pager`  
- 确认 **`data_directory`** 路径存在且 **`postgres:postgres`** 属主正确  
- 确认 **`/data`** 已挂载（`df -h /data`）

### 10.4 Docker 相关

- **`db` 起不来、报 data directory 权限错误**：核对 **`PGDATA_HOST_PATH`** 目录属主是否为镜像内 `postgres` 用户的 UID（多为 **70**），见 [Docker 生产部署](#docker-生产部署推荐) 前置条件。  
- **`web` 一直 waiting / `api` unhealthy**：`docker compose ... logs api`；常见为连不上 `db` 或 **`JWT_SECRET` / `PG_PASSWORD`** 未在 `.env.docker.prod` 中设置（Compose 要求必填项）。  
- **与开发 Compose 混用**：勿在同一宿主机端口上同时跑开发 **`web:3000`** 与生产 **`web:80`** 以外的冲突；生产请固定使用 **`docker-compose.prod.yml`**。  
- **`docker build` / `apt-get` 报 502、Unable to fetch**：多为 Debian 官方源瞬时故障或网络策略导致，**直接重试构建**；仍失败时可换时段构建，或在能稳定访问镜像源的环境（如腾讯云内构建）执行 **`docker compose ... build`**。

### 10.5 拉镜像失败：`registry-1.docker.io` / `i/o timeout`

在国内云服务器上常见：**默认访问 Docker Hub 超时**，报错类似：

`failed to resolve reference "docker.io/library/...": ... dial tcp ... registry-1.docker.io ... i/o timeout`

**做法一（推荐）：为 Docker 配置镜像加速**

腾讯云 CVM 可使用官方文档中的内网加速地址（与其它镜像站可并列，**任一条失效可换**）：

```bash
sudo mkdir -p /etc/docker
sudo tee /etc/docker/daemon.json <<'EOF'
{
  "registry-mirrors": [
    "https://mirror.ccs.tencentyun.com",
    "https://docker.m.daocloud.io"
  ]
}
EOF
sudo systemctl daemon-reload
sudo systemctl restart docker
```

然后重试：

```bash
sudo docker run --rm hello-world
```

若你使用云厂商控制台提供的「镜像加速」专属地址，可把上述 JSON 中的地址**换成控制台给出的 URL**。

**做法二：单次指定镜像前缀（验证 Engine 是否正常）**

不改配置时，可先试从国内镜像拉 **hello-world**：

```bash
sudo docker run --rm docker.m.daocloud.io/library/hello-world:latest
```

**与本项目构建**：`docker compose ... build` 仍会拉 `node`、`postgres`、`nginx` 等；除配置 `registry-mirrors` 外，还可按手册 [镜像构建与国内源](#镜像构建与国内源) 使用 **`BASE_IMAGE=docker.m.daocloud.io/library/node:22-bookworm-slim`**；`postgres`、`nginx` 若仍超时，可在 `docker-compose.prod.yml` 中把 `image:` 改为可访问的镜像地址（需与官方镜像兼容）。
